# RealSourcing 5.0 — 开发路线图

> **文档版本：** v1.0  
> **最后更新：** 2026-02-27  
> **配套文档：** 本文档与 `ARCHITECTURE.md` 配合使用。每次 AI Vibe Coding 会话开始前，必须同时阅读两份文档。

---

## 总体原则

RealSourcing 5.0 是一个**长期迭代项目**，不追求快速上线，而是追求每个阶段都能交付真实可用的价值。开发采用 AI Vibe Coding 模式（通过 Manus 等 AI 工具驱动），每个任务都有明确的输入/输出定义，确保不同会话生成的代码能够无缝衔接。

**MVP 定义：** 能够跑通"海外获得客户/询盘"的完整链路——即 OpenClaw 在海外平台执行操作 → 潜在买家产生询盘 → 老板收到飞书通知 → 在 Commander 中确认回复 → OpenClaw 发出回复。

## Phase 0：基础设施搭建

**目标：** 建立所有后续开发依赖的基础环境，确保各模块可以独立开发和测试。

**预计周期：** 2-3 周

### 任务 0.1：阿里云环境配置

| 项目 | 规格 | 说明 |
|------|------|------|
| ECS（Commander 后端） | 2核4G，Ubuntu 22.04 | 部署 FastAPI 服务 |
| RDS PostgreSQL | 2核4G，PostgreSQL 15 | 主数据库 |
| Redis | 1G 标准版 | 会话缓存 + 限流 |
| RocketMQ | 标准版 | 消息队列 |
| OSS | 按量付费 | 产品图片存储 |

**输出：** 所有服务的连接字符串和访问密钥，记录在 `.env.example` 文件中（不含真实密钥）。

### 任务 0.2：Commander 后端骨架

**输入：** `ARCHITECTURE.md` 中的技术选型和目录结构规范

**开发内容：**
- FastAPI 项目初始化（含 CORS、异常处理中间件、健康检查接口）
- SQLAlchemy 模型定义（对应 `ARCHITECTURE.md` 第 4 节所有表）
- Alembic 迁移初始化
- JWT 认证模块（登录/刷新/登出）
- 多租户中间件（自动注入 `tenant_id`）
- 基础 CRUD 工具函数

**输出：** 可运行的 FastAPI 服务，`GET /health` 返回 `{"status": "ok"}`，所有数据库表已创建。

### 任务 0.3：OpenClaw 基础框架

**输入：** `ARCHITECTURE.md` 中的 OpenClaw 与 Commander 通信协议（第 6 节）

**开发内容：**
- Playwright 浏览器管理器（支持多账号并发，每账号独立 browser context）
- 代理 IP 管理模块（轮换策略，健康检测）
- RocketMQ Producer（事件推送到 Commander）
- HTTP 客户端（接收 Commander 下发的任务指令）
- 心跳上报模块（每 5 分钟向 Commander 报告实例状态）

**输出：** OpenClaw 基础框架可运行，能向 Commander 发送心跳事件，Commander 能正确记录实例状态。

### 任务 0.4：Commander 手机端骨架

**输入：** 现有原型代码（`apps/commander-h5/`），`ARCHITECTURE.md` 认证接口规范

**开发内容：**
- 将原型的 mock 数据替换为真实 API 调用（Axios + React Query）
- JWT 认证流程（登录页 → Token 存储 → 自动刷新）
- 全局错误处理（网络错误 Toast + 401 自动跳转登录）
- 飞书 Deep Link 处理（`realsourcing://inquiry/{id}` 跳转逻辑）

**输出：** 手机端能完成登录，询盘列表从真实 API 加载（此时后端返回空数据）。

## Phase 1：MVP 核心链路

**目标：** 跑通"询盘获取 → 通知推送 → 老板确认回复 → OpenClaw 发出"完整链路。

**预计周期：** 4-6 周

### 任务 1.1：LinkedIn 询盘监控（OpenClaw）

**输入：** LinkedIn 账号 Session（由客户提供），Commander 的 Webhook 端点地址

**开发内容：**
- LinkedIn 消息收件箱监控（Playwright 定时轮询，间隔 5-15 分钟随机化）
- 新消息识别（对比上次检查时间，过滤已处理消息）
- 消息内容解析（买家姓名/公司/国家/产品需求提取，使用 LLM）
- 置信度评分计算（渠道权重 30 + 内容质量 35 + 买家完整度 35）
- 推送新询盘事件到 RocketMQ

**输出：** LinkedIn 收到新消息后，5-15 分钟内在 Commander 后端数据库创建一条询盘记录。

### 任务 1.2：Commander 后端 — 询盘接收与通知

**输入：** RocketMQ 消息格式（`ARCHITECTURE.md` 第 6.1 节）

**开发内容：**
- RocketMQ Consumer（消费 `inquiry.created` 事件）
- 询盘创建逻辑（写入数据库，计算置信度，关联租户）
- 飞书消息卡片推送（`ARCHITECTURE.md` 第 7 节格式）
- 推送失败重试（最多 3 次，指数退避）

**输出：** OpenClaw 推送新询盘后，老板的飞书收到消息卡片通知，卡片显示买家信息和预估金额。

### 任务 1.3：Commander 手机端 — 询盘详情与报价

**输入：** `ARCHITECTURE.md` 第 5.2 节 API 接口，飞书多维表格 API 文档

**开发内容：**
- 询盘列表真实数据加载（含三维筛选：渠道/国家/金额）
- 询盘详情页真实数据（买家信息/置信度/原始消息）
- 飞书报价模板读取（根据租户行业选择对应模板，展示字段）
- 报价填写表单（价格/数量/交货期/跟进风格选择）
- 报价提交（调用 `POST /inquiries/{id}/quote`）

**输出：** 老板能在手机端看到真实询盘，填写报价后提交，数据库创建报价记录。

### 任务 1.4：OpenClaw — 接收回复指令并发出

**输入：** Commander 下发的任务指令格式（`ARCHITECTURE.md` 第 6.2 节）

**开发内容：**
- HTTP 服务端（接收 Commander 的任务下发）
- LinkedIn 消息发送（Playwright 操作，模拟人工发送节奏）
- 发送结果回调（成功/失败事件推送到 RocketMQ）
- 操作日志记录（写入 `agent_logs` 表）

**输出：** Commander 下发回复指令后，OpenClaw 在 LinkedIn 以老板账号发出消息，Commander 更新询盘状态为"已报价"。

### 任务 1.5：飞书报价单写入

**输入：** 飞书多维表格 API，租户的飞书 Token 配置

**开发内容：**
- 飞书 OAuth 授权流程（首次配置时引导老板授权）
- 报价单写入飞书多维表格（创建新记录，字段按行业模板映射）
- 写入成功后更新 `quotations.feishu_record_id`

**输出：** 报价发出后，飞书多维表格自动新增一条报价记录，包含买家信息和报价详情。

## Phase 2：询盘管理完整闭环

**目标：** 完善询盘全生命周期管理，实现自动跟进和状态流转。

**预计周期：** 3-4 周

### 任务 2.1：买家回复监控与通知

**开发内容：**
- OpenClaw 监控已报价询盘的买家回复（LinkedIn/WhatsApp）
- 买家回复事件推送到 RocketMQ（`inquiry.replied`）
- Commander 后端处理回复事件（更新询盘状态，推送飞书通知）
- 手机端展示买家回复内容，支持继续对话

### 任务 2.2：24小时自动跟进

**开发内容：**
- 报价发出后，Commander 后端创建 `followup_tasks` 记录（scheduled_at = 发出时间 + 24h）
- RocketMQ 延迟消息（24h 后触发 `followup.trigger` 事件）
- OpenClaw 接收跟进指令，按选定风格（强势/友好/商务）发送预制话术
- 跟进后更新时间线，若买家仍未回复则询盘状态变为"客户未回复"

### 任务 2.3：询盘状态机完整实现

**开发内容：**
- 后端状态流转验证（防止非法状态跳转，如从"已转合同"退回"未报价"）
- 转人工业务员（飞书通知指定业务员，附带询盘摘要）
- 询盘失效归档（批量操作支持）
- 询盘统计 API（各状态数量、转化率、平均响应时间）

### 任务 2.4：每日战报生成与推送

**开发内容：**
- 定时任务（北京时间 08:00 和 18:00）生成当日战报
- 战报内容：新增询盘数/已回复数/报价金额合计/OpenClaw 操作次数/积分消耗
- 飞书消息卡片格式推送（与新询盘通知区分样式）
- 手机端今日状态视图从真实 API 加载战报数据

## Phase 3：社媒账号托管

**目标：** 实现 LinkedIn/Facebook/TikTok/WhatsApp 四平台的账号托管，评论和私信统一管理。

**预计周期：** 4-6 周

### 任务 3.1：账号 Session 管理

**开发内容：**
- 账号 Session 加密存储（AES-256，存储在数据库）
- Session 有效性检测（定时检查，失效时推送告警通知）
- 账号健康监控（操作次数统计，接近阈值时预警）
- 代理 IP 与账号绑定（每个账号固定使用特定代理节点）

### 任务 3.2：LinkedIn 完整托管

**开发内容：**
- 连接请求管理（接受/忽略，批量处理）
- InMail 私信监控（新消息识别，推送到询盘管理）
- 评论监控（识别询价评论，推送到询盘管理）
- 账号操作额度控制（每日私信 < 20条，连接请求 < 30个）

### 任务 3.3：Facebook 个人主页托管

**开发内容：**
- 私信监控（Messenger 新消息识别）
- 评论监控（帖子评论中的询价识别）
- 自动回复开关（私信可开启全自动，评论需人工确认）
- 操作额度控制

### 任务 3.4：TikTok 企业账号托管

**开发内容：**
- TikTok Content API 集成（官方 API，需企业账号）
- 评论监控（识别 "how much" / "price" / "MOQ" 等询价关键词）
- 评论回复（人工确认后发出）
- 数据指标同步（播放量/完播率/互动率，每日同步）

### 任务 3.5：WhatsApp Business 托管

**开发内容：**
- WhatsApp Business API 集成（或 Playwright 模拟，取决于账号类型）
- 对话列表监控
- 快捷回复模板管理
- 自动回复开关

## Phase 4：多 Agent 协同与高级功能

**目标：** 实现新市场开发、GEO 优化、产品发布等高级 Agent 任务，以及积分计费系统。

**预计周期：** 持续迭代

### 任务 4.1：新市场开发计划 Agent 链

**开发内容：**
- 调研 Agent（目标市场数据收集：市场规模/主要平台/竞争对手）
- 内容 Agent（本地化营销内容生成：语言/文化适配）
- 外联 Agent（OpenClaw 执行 LinkedIn/Facebook 触达）
- 监控 Agent（指标追踪，每日战报）
- 多 Agent 协调器（任务依赖管理，状态同步）
- 手机端进度追踪（各 Agent 当前状态可视化）

### 任务 4.2：GEO 优化 Agent

**开发内容：**
- 目标 AI 搜索引擎选择（Perplexity/ChatGPT/Gemini/Claude/Grok）
- 内容优化策略生成（针对 AI 引用偏好的内容结构）
- 内容发布执行（博客/问答/行业目录）
- 引用监控（定期检查是否被 AI 引用，统计曝光量）

### 任务 4.3：新产品发布 Agent

**开发内容：**
- 产品信息录入（名称/卖点/目标市场/图片上传到 OSS）
- 多语言内容生成（LLM 生成英/西/阿/越南语等描述）
- 多平台发布（LinkedIn 帖子/Facebook 帖子/TikTok 视频脚本）
- 发布状态追踪

### 任务 4.4：积分计费系统

**开发内容：**
- 积分消耗规则定义（按任务类型和资源消耗计算）
- 积分余额实时更新（任务完成后扣减）
- 积分消耗记录（详细日志，支持按时间段查询）
- 积分不足预警（余额低于阈值时推送飞书通知）
- 积分充值接口（对接支付系统）

### 任务 4.5：语音助手（真实接入）

**开发内容：**
- 后端语音转文字（Whisper API，服务器处理）
- 指令意图识别（LLM 解析语音内容，提取操作意图和参数）
- 安全操作直接执行 / 敏感操作返回预览等待确认
- 批量指令处理（一句话处理多条询盘）

## Phase 5：React Native 正式 App

**目标：** 将 PWA 原型迁移到 React Native，提供原生 App 体验。

**预计周期：** 6-8 周

### 关键迁移任务

| 任务 | 说明 |
|------|------|
| 业务逻辑复用 | 将 React 中的 hooks、状态管理、API 调用层迁移到 React Native |
| 原生推送通知 | 接入 APNs（iOS）和 FCM（Android），替代飞书通知作为 App 内通知 |
| Deep Link | 配置 `realsourcing://` URL Scheme，支持从飞书通知跳转到 App |
| 语音输入 | 使用 React Native 原生麦克风 API，替代 Web Audio API |
| App Store 上架 | iOS App Store + Google Play 双平台发布 |

## AI Vibe Coding 会话指南

每次开始新的 Manus 开发会话时，请按以下步骤操作：

**第一步：** 告知 AI 当前要完成的任务编号（如"完成 Phase 1 任务 1.1"）

**第二步：** 让 AI 读取以下文件：
- `docs/ARCHITECTURE.md`（技术架构和规范）
- `docs/ROADMAP.md`（当前任务的输入/输出定义）
- 相关模块的现有代码（如有）

**第三步：** 确认 AI 理解任务的输入/输出定义后，开始开发

**第四步：** 完成后更新本文档中的"当前状态"表格

> **重要提示：** 每个任务完成后，必须运行自动化测试验证输出是否符合定义。AI 生成的代码容易在边界情况下出错，测试是保证质量的唯一手段。
