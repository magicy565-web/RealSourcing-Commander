# RealSourcing Commander “以假乱真”MVP 作战手册 (V2.3)

> **文档目标**：基于“购买 Kalodata/Fastmoss 会员 + 定时脚本上传报告 + LLM 提取分析”的核心思路，设计一套技术上最简单、商业上最讨巧的“Wizard of Oz” MVP 方案。本文档将详细阐述其数据流水线、技术实现、成本结构与风险边界，为单人创业者提供一套剑走偏锋、快速验证市场的行动指南。

**更新日期**：2026年3月4日
**核心原则**：**在用户看不到的地方，用最“笨”的办法解决问题；在用户能看到的地方，把体验做到极致。**

## 1. 调研结论：Kalodata 与 Fastmoss 的能力边界

经过对公开资料、评测文章和社区讨论的综合分析，我们得出以下核心结论：

### 1.1. 两者的共同点与差异

| 对比维度 | Kalodata | Fastmoss | 结论 |
| :--- | :--- | :--- | :--- |
| **核心定位** | **战术狙击镜** | **战略雷达** | Kalodata 适合找“当下爆款”，Fastmoss 适合看“未来趋势”。 |
| **数据刷新率** | 15 分钟 | 每日 | Kalodata 的实时性更强，更符合我们“爆款雷达”的需求。 |
| **数据导出** | **支持 CSV 导出** | **支持 CSV 导出** | 两者均支持将榜单数据（如达人、商品、视频）导出为 CSV 文件，这是我们方案的技术基石。 |
| **数据维度** | 侧重商品和达人 ROI | 侧重跨国市场和品类增长 | 两者数据维度互补，长期来看可以都用。 |
| **价格** | Pro 版 ~$99/月 | Pro 版 ~$109/月 | 价格相近，对于 MVP 阶段可接受。 |

### 1.2. 关键发现：数据源于“抓取”而非“官方 API”

最重要的一点是，根据 Dashboardly 的对比评测，**Kalodata 和 Fastmoss 的核心数据都来源于大规模的 Web Scraping（网页抓取），而非 TikTok 的官方 API**。这带来了三个关键影响：

1.  **数据并非 100% 精准**：抓取数据在 GMV、销量等指标上可能存在偏差，不应作为财务依据，但作为“趋势”和“情报”分析已足够。
2.  **数据字段可能变更**：由于依赖抓取，一旦 TikTok 前端页面结构变化，导出的 CSV 文件字段也可能随之改变，我们的解析脚本需要具备一定的健壮性。
3.  **法律与合规风险**：依赖抓取数据存在一定的法律灰色地带。我们需要在服务条款中明确说明，数据仅供参考，不作为商业决策的唯一依据。

### 1.3. 最终选型：MVP 阶段优先选择 Kalodata

基于以上分析，我们 MVP 阶段的策略是：

- **优先订阅 Kalodata 的 Pro 会员**。理由是其 15 分钟的刷新率和对“实时爆款”的侧重，更符合我们向用户提供的核心价值——“快人一步发现商机”。
- **利用其 CSV 导出功能**。这是我们整个“以假乱真”方案的技术前提。根据知乎和 YouTube 的信息，其导出功能强大，可以一次性导出上万条数据。

有了这个明确的输入源，我们现在可以设计完整的数据流水线。

## 2. 数据流水线设计：“人工”+“自动”的混合双打

这个方案的精髓在于，将最复杂、最不可控的“数据抓取”环节替换为**“人工下载上传”**，而将重复性、可标准化的“数据处理与分析”环节实现自动化。

### 2.1. 数据流架构图

```mermaid
graph TD
    A[1. 人工操作: 登录 Kalodata] --> B(2. 人工操作: 下载行业报告 CSV);
    B --> C{3. 人工操作: 上传 CSV 到 S3/Supabase Storage};
    C --> D[4. 自动触发: Vercel Cron Job (每天一次)];
    D --> E[5. 自动执行: Serverless Function];
    E --> F{6. 数据处理: 读取新 CSV 文件};
    F --> G[7. AI 分析: 调用 LLM API 分析每一行数据];
    G --> H[8. 数据落库: 将分析结果写入 Supabase 数据库];
    H --> I[9. 前端展示: 用户在产品中看到分析结果];
```

### 2.2. 各环节详解

#### 步骤 1-3: “你”就是那个最可靠的爬虫 (人工操作)

- **频率**：每天一次，或每周三次，根据你的目标客户对数据新鲜度的要求而定。
- **操作**：
  1.  登录你的 Kalodata Pro 会员账号。
  2.  根据你为产品设定的目标行业（如“户外装备”、“美妆个护”），分别进入对应的榜单页面（如“热卖商品榜”、“飙升达人榜”）。
  3.  点击“导出 CSV”按钮，将数据下载到本地。
  4.  将下载的 CSV 文件，通过 Supabase 的管理后台，手动上传到指定的 Storage Bucket 中。为了便于脚本处理，建议文件名遵循固定格式，如 `kalodata_products_outdoor_2026-03-04.csv`。

- **时间成本**：整个过程熟练后，每天耗时**不超过 15 分钟**。

#### 步骤 4-8: “代码”负责所有重复劳动 (自动执行)

这是我们需要编写代码的部分，但全部基于 Serverless，没有运维负担。

1.  **定时触发 (Vercel Cron Job)**：
    - 在 Vercel 的项目设置中，添加一个 Cron Job，设置为每天凌晨 4 点（或你手动上传完数据后的任何时间）触发。
    - Cron Job 的目标是一个 HTTP GET 请求，指向我们部署在 Vercel 上的一个 Serverless Function，例如 `/api/tasks/process-kalodata-reports`。

2.  **核心处理函数 (Serverless Function)**：
    - 这个函数是整个自动化的核心，它会执行以下逻辑：
      - **列出新文件**：使用 Supabase 的 Node.js 客户端，列出 Storage Bucket 中当天新增的所有 CSV 文件。
      - **逐行读取与解析**：使用 `papaparse` 或类似的库，以流式（streaming）方式读取 CSV 文件内容，避免因文件过大导致内存溢出。
      - **调用 LLM 分析**：对于 CSV 中的每一行数据（代表一个商品或一个达人），构造一个精心设计的 Prompt，调用 OpenAI 或其他 LLM 的 API，提取关键信息或进行二次分析。
        - **示例 Prompt (分析商品)**：
          ```
          Analyze the following TikTok product data from a CSV row and return a JSON object.
          Data: {csv_row_json}
          
          Based on the data, perform the following analysis:
          1.  `sales_pitch`: Write a 3-sentence compelling sales pitch for this product, targeting its likely audience.
          2.  `target_audience`: Describe the target audience in 3-5 keywords.
          3.  `content_angle`: Suggest 3 creative video content angles to promote this product.
          4.  `risk_assessment`: Briefly assess the market saturation risk (Low, Medium, High).
          
          Return only the valid JSON object.
          ```
      - **数据落库**：将原始的 CSV 行数据，连同 LLM 返回的分析结果 JSON，一并存入 Supabase 的对应数据表（如 `products`, `creators`）中。
      - **标记已处理**：处理完一个 CSV 文件后，将其移动到另一个“已处理”的 Bucket 或重命名，避免下次被重复处理。

### 2.3. 前端展示

- 你的 React 前端页面，现在可以直接通过 Supabase 的客户端库，或其自动生成的 REST API，安全地查询 `products` 表中的数据。
- 用户看到的将是经过 LLM “精加工”后的、富有洞察力的情报，而不是冷冰冰的原始数据。他们完全不知道，这些数据的源头是你每天早上花了 15 分钟手动下载的。

这个方案的巧妙之处在于，它将**人的灵活性**（应对 Kalodata 网站的任何变化）与**机器的效率**（处理数万行数据的重复劳动）完美结合，用最小的开发成本，实现了看似非常智能和自动化的产品效果。

## 3. 成本结构、风险边界与合规性

### 3.1. 月度成本核算

| 成本项目 | 方案 | 月度费用 | 备注 |
| :--- | :--- | :--- | :--- |
| **数据源** | Kalodata Pro 会员 | ~$99 | 核心成本，提供原始数据 |
| **LLM 分析** | OpenAI gpt-4.1-nano | ~$5-15 | 假设每天处理 500 行数据，每行约 500 tokens，每月约 750 万 tokens，nano 模型成本极低 |
| **数据库** | Supabase 免费套餐 | $0 | 500MB 存储，50MB 数据库，足够 MVP 阶段 |
| **后端 API** | Vercel 免费套餐 | $0 | 100GB 带宽，100 万次 Function 调用 |
| **前端托管** | Vercel 免费套餐 | $0 | 与后端同项目 |
| **合计** | | **~$104-114/月** | 这是你的"数据成本"，需要通过产品定价覆盖 |

**关键结论**：只要你的产品能吸引到**3-4 个付费用户**，每人每月付费 $39-49，就能覆盖全部成本并开始盈利。这是一个极低的盈亏平衡点。

### 3.2. 风险边界分析

这个方案存在三个需要清醒认识的风险：

**风险一：Kalodata 封号或限制导出功能（中风险）**

Kalodata 的服务条款中可能限制将其数据用于商业转售。如果他们发现你的账号在大量导出数据，可能会封禁账号。

**应对策略**：
- 注册时使用个人名义，而非公司名义，降低被关注的可能性。
- 导出量保持克制，不要每次都导出最大量，模拟普通用户行为。
- **最重要**：将这个方案定位为"过渡性的"。当你的产品有了稳定收入后，立即开始开发自己的数据采集能力，或与 Kalodata 洽谈官方数据合作。

**风险二：数据准确性问题（低风险，可管理）**

如前所述，Kalodata 的数据本身来自抓取，存在偏差。

**应对策略**：
- 在产品的 UI 中，以小字注明"数据来源于第三方情报平台，仅供参考，不作为商业决策的唯一依据"。
- 这不仅是合规要求，也是合理的用户预期管理。

**风险三：LLM 分析质量不稳定（低风险，可管理）**

LLM 对 CSV 数据的分析质量，高度依赖 Prompt 的设计。如果 Prompt 不好，输出的"洞察"可能是废话。

**应对策略**：
- 在上线前，花时间对 Prompt 进行迭代优化，至少测试 50 条数据。
- 在前端 UI 上，提供"这个分析有帮助吗？"的反馈按钮，收集用户反馈，持续改进 Prompt。

### 3.3. 这个方案的"天花板"在哪里？

这个方案最大的局限性是**可扩展性**。当你的用户量增长到一定程度，你会遇到以下瓶颈：

- **数据新鲜度**：每天手动上传一次，意味着数据最多是"昨天"的，无法做到实时。
- **行业覆盖度**：你每天能手动下载的行业报告数量是有限的，无法覆盖所有行业。
- **人工成本**：随着用户需求增加，每天 15 分钟可能会变成 1 小时。

**这些恰恰是你未来的"护城河"**。当你有了第一批付费用户和稳定收入后，你就知道哪些行业是用户最需要的，然后有针对性地开发自动化采集能力（即 Agent 01/02 的真实版本），逐步替换掉手动环节。这才是正确的产品演进路径。

## 4. 总结：这是一个"正确"的 MVP 策略

你提出的这个方案，在创业方法论上有一个专业的名字：**"Wizard of Oz" MVP**——在用户眼中，产品是一个强大的 AI 自动化系统；而在幕后，是你这个"巫师"在手动操控。

这种策略被大量成功的 SaaS 公司在早期使用过。它的核心价值在于：

1.  **极低的开发成本**：你不需要解决爬虫、反封控等最难的技术问题，就能向用户交付价值。
2.  **快速验证市场**：你可以在 2-3 周内上线，而不是 2-3 个月。
3.  **真实的用户反馈**：你能快速知道用户最需要哪个行业的数据、最看重哪个分析维度，从而指导后续的技术投入方向。

**建议的行动步骤**：
1.  **本周**：订阅 Kalodata Pro，手动下载 3 个行业的数据，设计并测试 LLM 分析 Prompt。
2.  **第 2 周**：开发 Supabase 数据库 Schema + Vercel Cron Job + 处理函数。
3.  **第 3 周**：在 Commander 的前端，新增一个"行业情报"页面，展示分析结果。
4.  **第 4 周**：找到 5 个目标用户，免费让他们试用，收集反馈。
5.  **第 5 周起**：根据反馈迭代，开始向用户收费。
