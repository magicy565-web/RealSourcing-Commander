# RealSourcing Commander 单人创业者作战手册 (V2.2)

> **文档目标**：在“只有一个人”的核心约束下，重新制定 AI 全家桶的商业化落地路线图。本文档将彻底抛弃需要团队维护的复杂工程实践，以“最小可行产品 (MVP)”为核心，全面转向“零运维”的 Serverless 和托管服务，为单人创业者提供一份务实、低风险、可执行的行动指南。

**文档作者**：Manus AI
**更新日期**：2026年3月4日
**核心原则**：**如果一个技术需要你 `ssh` 到服务器上进行任何操作，那就果断放弃它。**

## 1. 重新评估技术栈：用“托管服务”替代“自建”，消灭一切运维负担

对于单人开发者，最大的成本不是金钱，而是**时间**和**认知负荷**。V2.1 文档中的微服务架构虽然技术上完美，但对个人而言是“不可承受之重”。我们需要一个全新的、以“零运维”为目标的 Serverless 优先技术栈。

### 架构对比：从“理想国”到“现实世界”

| 组件 | V2.1 团队版架构 (理想国) | V2.2 单人版架构 (现实世界) | 核心理由 |
| :--- | :--- | :--- | :--- |
| **Web 前端** | Vite + React (部署在 Vercel) | **Vite + React (部署在 Vercel)** | **(不变)** Vercel 的前端托管和 CI/CD 是行业最佳实践，继续沿用。 |
| **API 服务** | Node.js on Kubernetes | **Next.js API Routes** 或 **Hono on Vercel** | 将后端 API 直接放在 Vercel 的 Serverless Functions 中。**零配置、自动扩缩容、与前端天然集成。** |
| **数据库** | 自建 PostgreSQL | **Supabase** 或 **Neon** | **(关键变更)** Supabase 是“带电池的 PostgreSQL”，它不仅提供全托管数据库，还**免费附赠了用户认证、对象存储、甚至自动生成 REST API** 的功能，能帮你节省数周的开发时间。 |
| **任务队列** | Redis + BullMQ | **Upstash QStash** | **(关键变更)** 这是一个专为 Serverless 设计的 HTTP 任务队列。你只需调用一个 API 就能将任务推送到队列，它会在指定时间后通过 Webhook 回调你的另一个 Serverless Function。**彻底告别 Redis 服务器的部署、监控和维护。** |
| **爬虫 Worker** | Python + Playwright on K8s | **第三方抓取 API** | **(最关键的变更)** 放弃自建爬虫！这是单人开发者最大的时间黑洞。将这个最脏最累的活外包出去。 |
| **可观测性** | ELK + Jaeger + Sentry | **Vercel Logs + Sentry** | Vercel 自带实时日志查看。Sentry 的免费套餐足以满足 MVP 阶段的错误监控需求。**简单、够用，无需部署。** |

### 新架构下的工作流示例 (Agent 01)

1.  用户在 Vercel 托管的前端页面上点击“开始扫描”。
2.  请求发送到同一个 Vercel 项目下的 `/api/agents/trigger` Serverless Function。
3.  该函数不执行任何耗时操作，而是直接调用 **Upstash QStash** 的 API，将任务参数（目标账号）推送到队列，并设置一个 5 秒后的回调 URL，指向 `/api/agents/worker`。
4.  函数立即向前端返回 `task_id`，UI 显示“任务已启动”。
5.  5 秒后，QStash 调用 `/api/agents/worker` 函数。
6.  这个 Worker 函数调用 **Bright Data 的 Scraping Browser API**，将抓取任务外包出去。这是一个简单的 `fetch` 调用。
7.  Bright Data 在后台处理所有复杂的代理、指纹和验证码问题，几分钟后返回抓取到的评论 JSON 数据。
8.  Worker 函数拿到数据，调用 LLM 进行分析，然后使用 **Supabase** 的客户端库将结果写入 `leads` 表。

在这个流程中，你**没有管理一台服务器，没有配置一个 Docker 容器，没有维护一个 Redis 实例**。你的所有代码都只是在不同的 Serverless Function 之间通过 HTTP 调用传递数据，这才是单人开发者能够驾驭的模式。

## 2. 单人 MVP 路线图：从“做加法”到“做减法”

12 个 Agent 一起上是不现实的。单人创业的核心是**“找到一个最小但有价值的闭环，快速验证市场”**。我们需要对 Agent 进行优先级重排，并分阶段交付。

### 2.1. 优先级重排：寻找“一针捅破天”的单点

对于外贸客户，最痛的点永远是**“获取高质量询盘”**。因此，我们的 MVP 应该 100% 聚焦于此。其他如内容生成、财务管理等，都是“有了更好”但“没有也行”的功能。

**新优先级排序**：

| 优先级 | Agent | 核心价值 (一句话) | 为何优先 |
| :--- | :--- | :--- | :--- |
| **P0 (核心)** | **01 线索猎手** | 帮你找到想买东西的客户。 | **(直接价值)** 这是整个产品的价值主张，是客户付费的根本原因。 |
| **P0 (核心)** | **08 邮件跟进** | 帮你自动联系这些客户。 | **(价值闭环)** 找到线索却不联系，价值为零。这是形成最小闭环的必要一环。 |
| **P1 (增强)** | **07 私信客服** | 帮你 24h 自动回复初步意向。 | 在线索量大时，能有效过滤低意向客户，提升销售效率。 |
| **P2 (扩展)** | **02 爆款雷达** | 告诉你做什么内容能火。 | 从“找客户”扩展到“吸引客户”，是产品价值的自然延伸。 |
| **P2 (扩展)** | **03 选题助手** | 帮你写出能火的内容脚本。 | 与 Agent 02 配合，形成内容策略闭环。 |
| **P3 (锦上添花)** | 04, 05, 06, 09-12 | 其他所有 Agent | 这些功能很好，但都可以在 MVP 阶段被**手动操作**或**第三方成熟软件**（如 Stripe, Xero）替代。 |

### 2.2. 三阶段交付路线图

#### 阶段一：MVP — “线索获取与跟进”最小闭环 (预计 1-2 个月)

**目标**：只做 Agent 01 和 Agent 08，让客户能实实在在地收到“高意向询盘邮件列表”。

- **Agent 01 (线索猎手)**：
  - **放弃自建爬虫**，全面转向 Bright Data 或 Apify 等第三方抓取 API。
  - 前端允许用户输入 1-3 个竞品 TikTok/Instagram 账号。
  - 后端通过 QStash 触发 Worker，调用抓取 API 获取评论。
  - 使用 `gpt-4.1-mini` 分析意图，将高意向线索存入 Supabase。
- **Agent 08 (邮件跟进)**：
  - **放弃自建邮件序列**，全面转向 **Resend** 或 **Mailgun** 的 API。
  - 提供一个简单的三步邮件序列模板（介绍 -> 案例 -> 追问）。
  - 用户在 UI 上填写自己的 API Key 和模板内容。
  - Worker 从 Supabase 读取新线索，通过 Resend API 自动发送邮件。

**交付物**：一个能让客户每天早上收到“昨日新增高意向线索及已发送邮件报告”的 SaaS 产品。

#### 阶段二：增强 — “自动化互动与内容洞察” (MVP 上线后 2-3 个月)

**目标**：在 MVP 验证市场后，加入 Agent 07, 02, 03，提升产品护城河。

- **Agent 07 (私信客服)**：
  - **放弃自建 RAG**，使用 **Vercel AI SDK + OpenAI Assistants API**。Assistants API 内置了 RAG 和多轮对话管理，能极大降低开发复杂度。
- **Agent 02/03 (爆款雷达/选题助手)**：
  - 同样使用第三方抓取 API 获取竞品视频数据。
  - 核心是设计好 Prompt，让 LLM 做好分析和脚本生成。

**交付物**：产品从“线索工具”升级为“社媒营销情报平台”。

#### 阶段三：完善 — “后端自动化” (持续迭代)

**目标**：在产品获得稳定收入后，再考虑将 P3 梯队的 Agent 逐一实现，以“降本增效”为主要卖点。

- **Agent 04 (数字分身)**：集成 HeyGen API。
- **Agent 09 (收单收款)**：集成 Stripe Webhook。

这个路线图的核心是**“价值驱动，逐步替换”**。在早期，用最快的、最“脏”的方式（外包、手动）实现核心价值闭环；在后期，再用更优雅、更自动化的方式，逐步替换掉那些“脏”的部分，优化成本和效率。

## 3. 单人作战手册：三大难题的“零运维”解决方案

针对 V2.1 文档中提出的三大核心工程挑战，这里给出一套单人开发者可以实际操作的解决方案。

### 3.1. 爬虫维护 -> 全面外包给 API

- **挑战**：24 小时监控和维护 Playwright 脚本，对抗平台反爬。
- **单人解决方案**：**坚决不自己写爬虫。** 这是单人开发者最容易掉进去的陷阱。
  - **首选服务**：**Bright Data 的 Scraping Browser API** 或 **Apify 的 TikTok Scraper Actor**。
  - **成本分析**：以 Bright Data 为例，抓取 1000 个页面的成本大约是 3-5 美元。假设一个目标账号每天需要抓取 10 个视频的评论（每个视频算 1 个页面），扫描 10 个目标账号，每天的成本是 `10 * 10 * (3/1000) = 0.3` 美元。一个月下来不到 10 美元。**用 10 美元/月，换取你每周至少 10 小时的维护时间，这笔交易无比划算。**
  - **代码实现**：你的代码将从复杂的 Playwright 脚本，简化为下面这样一段简单的 `fetch` 调用：
    ```typescript
    const response = await fetch('https://api.brightdata.com/...', {
      method: 'POST',
      headers: { 'Authorization': 'Bearer YOUR_API_KEY' },
      body: JSON.stringify({ url: 'https://www.tiktok.com/@some_user/video/...' })
    });
    const data = await response.json();
    // ... process data
    ```

### 3.2. 可观测性 -> 依赖 PaaS 平台自带能力

- **挑战**：搭建和维护 ELK, Jaeger, Sentry 等复杂监控系统。
- **单人解决方案**：**充分利用你所选择的 PaaS 平台提供的免费或低价服务。**
  - **日志**：**Vercel Logs**。所有部署在 Vercel 上的 Serverless Function 的 `console.log` 输出都会自动汇集到 Vercel 的日志浏览器中，支持实时查看和搜索。对于 MVP 阶段足够了。
  - **错误告警**：**Sentry**。Sentry 提供一个慷慨的免费套餐，你只需在你的 Next.js 应用中引入它的 SDK，所有未捕获的前后端异常都会被自动上报到 Sentry 后台，并可以通过邮件或 Slack 通知你。配置过程不超过 10 分钟。
  - **分布式追踪**：**暂时放弃**。在 MVP 阶段，你的服务调用链非常简单（前端 -> API Route -> QStash -> Worker -> Supabase），通过 Vercel Logs 和 Sentry 已经可以定位大部分问题。等到业务复杂后再考虑引入更专业的工具。

### 3.3. 降级手册 -> 简化为“重试与告警”

- **挑战**：为每个外部依赖设计复杂的降级策略和熔断器。
- **单人解决方案**：在 MVP 阶段，我们不需要复杂的断路器模式。核心目标是**“不要让一次性的网络抖动杀死我的任务”**和**“当严重问题发生时我能第一时间知道”**。
  - **核心工具**：**Upstash QStash 的自动重试机制**。
  - **实现**：
    1.  在创建 QStash 任务时，设置重试策略，例如 `retries: 3`。
    2.  你的 Worker 函数 (`/api/agents/worker`) 必须是**幂等**的。这意味着即使它被重复调用三次，也只会产生一次的业务效果（例如，通过检查数据库中是否已存在该线索来避免重复插入）。
    3.  如果 Worker 函数因为第三方 API（如 Bright Data, OpenAI）返回 5xx 错误而抛出异常，QStash 会自动在几分钟后重新调用它，最多重试 3 次。
    4.  如果重试 3 次后仍然失败，QStash 会将这个“死信”任务发送到一个你预先配置好的“死信队列 (Dead-Letter Queue)” Webhook 地址。
    5.  这个 Webhook 地址指向一个新的 Serverless Function，`/api/tasks/failed`。这个函数的唯一职责，就是**使用 Sentry 手动捕获一个严重错误，并附上失败的任务参数**。
    ```typescript
    import * as Sentry from "@sentry/nextjs";

    export default function handler(req, res) {
      const failedTask = req.body;
      Sentry.captureException(new Error('Task failed after 3 retries'), {
        extra: { task: failedTask },
        level: 'fatal',
      });
      res.status(200).send('OK');
    }
    ```
  - **效果**：通过这个简单的流程，你实现了一个轻量但有效的降级系统。临时性错误会被自动重试解决，而持久性问题（如 API Key 失效、平台规则大改）会以“Fatal”级别推送到你的 Sentry 告警中，让你能立即介入处理。

## 结语：单人创业者的胜利之路

作为单人创业者，你的最大优势是**速度**和**灵活性**。V2.2 的路线图正是为了最大化这两个优势而设计。它牺牲了 V2.1 架构的长期可扩展性和性能，但换来了 MVP 阶段最宝贵的资源：**极低的开发和运维成本，以及聚焦于核心价值验证的能力。**

请记住，商业产品的成功，始于一个能解决真实痛点的、哪怕是粗糙的 MVP。先用这套“零运维”的务实方案活下来，等到产品获得市场验证、带来稳定现金流后，你才有资本去招聘团队，再去追求 V2.1 中那个更宏大、更健壮的工程理想架构。
